# cimtool-cli

A standalone Maven project that assembles the CIMTool command-line interface into
a single self-contained executable uber JAR (`cimtool-cli.jar`) using the
Maven Shade plugin. While the CIMTool Eclipse RCP application is currently
released for Windows only — with macOS and Linux under evaluation — `cimtool-cli.jar`
has no Eclipse or UI dependencies and can be run headless on any platform where
Java 11 or later is available.

No Eclipse, OSGi runtime, or workspace is required to run the resulting JAR.



## Overview

This project is a **packaging-only** Maven module. It contains no Java source of
its own. Its sole purpose is to:

1. Collect `kena.jar` and `cimutil.jar` from a CIMTool Eclipse PDE product export
2. Collect all vendored third-party JARs from `Kena/lib/` and `CIMUtil/lib/`
3. Embed the XSLT builder stylesheets and Java builder configurations from `CIMUtil/builders/` at the classpath root
4. Merge everything into a single executable uber JAR via Maven Shade

The CLI entry point is `au.com.langdale.profiles.cl.CIMToolCLI`, which lives in
`CIMUtil/src/`.

### Eclipse PDE Product Export Dependency

This project sources `kena.jar` and `cimutil.jar` from a CIMTool Eclipse PDE
product export. These JARs are **not** generated by Maven and are **not** taken
from the Eclipse workspace `bin/` directories — they must come from a full PDE
product export performed as described in `CIMToolProduct/CIMToolProduct-README`
under the **PDE Product Export** section.

After the export completes, the `plugins\` directory under the export root will
contain versioned plugin folders including:

```
<export-root>\plugins\au.com.langdale.kena_<version>\kena.jar
<export-root>\plugins\au.com.langdale.cimutil_<version>\cimutil.jar
```

The install script locates these JARs automatically by searching for plugin folders
matching `au.com.langdale.kena_*` and `au.com.langdale.cimutil_*` and extracts the
version number from the plugin folder name (e.g. `au.com.langdale.kena_3.4.0` → `3.4.0`).
This means the script works for any release version without modification.

> **Note:** The version extracted from the plugin folder name must match the `<version>`
> declared for `kena` and `cimutil` in `pom.xml`. Update `pom.xml` when the
> version changes.

## Dependency Management — Local File Repository

### Why a Local Maven File Repository?

All of this project's dependencies are vendored JARs that live under `Kena/lib/`
and `CIMUtil/lib/` in the Eclipse PDE project tree, plus `kena.jar` and
`cimutil.jar` sourced from the CIMTool Eclipse PDE product export.
None of these are published to Maven Central.

Maven's `system` scope (the most obvious way to reference local JARs) is formally
deprecated as of Maven 3.8 and causes a false-positive validation error in Eclipse's
embedded Maven (m2e):

```
'dependencies.dependency.systemPath' for com.hp.hpl.jena:jena:jar must specify
an absolute path but is ${project.basedir}/../Kena/lib/jena-2.6.3.jar
```

To avoid this, the project uses a **local file repository** instead. A `lib-repo/`
directory inside `cimtool-cli/` is declared as a standard Maven repository in
`pom.xml`:

```xml
<repositories>
    <repository>
        <id>local-lib-repo</id>
        <url>file://${basedir}/lib-repo</url>
    </repository>
</repositories>
```

All dependencies are then declared as normal Maven dependencies with no `system`
scope or `systemPath` — m2e resolves them cleanly against `lib-repo/` with no
errors or warnings.

### Populating lib-repo/

> **Release context:** For official releases, `install-jars.bat` is invoked
> automatically by `release-cimtool.bat` — no manual action is needed.
> The instructions below apply to **local development only**.

The `lib-repo/` directory is populated by running the provided install script, which
calls `mvn install:install-file` for each JAR and places it into `lib-repo/` using
the standard Maven repository layout (`groupId/artifactId/version/artifact-version.jar`).

**Passing the export root as an argument:**
```
cd cimtool-cli
install-jars.bat D:\CIMTool-Releases\CIMTool-2.3.0
```

**Path contains spaces — wrap in double quotes:**
```
cd cimtool-cli
install-jars.bat "D:\My Releases\CIMTool-2.3.0"
```

**Omit the argument to be prompted interactively:**
```
cd cimtool-cli
install-jars.bat
```

When run without an argument the script prompts:
```
  No export root specified.
  Enter the path to the CIMTool Eclipse PDE product export root.
  Example: D:\CIMTool-Releases\CIMTool-2.3.0
  Tip: If the path contains spaces, wrap it in double quotes.

  Export root: _
```

After installing the two core module JARs, the script prompts whether to
also install the vendor JARs:

```
  The vendor JARs (Kena\lib and CIMUtil\lib) rarely change between
  releases. Re-installing them is only needed if new or updated JARs
  have been added to those directories since the last run.
  If unsure, answer Y to ensure lib-repo is fully up to date.

  Install vendor JARs? (Y/N): _
```

The script processes all 26 dependencies in three groups:

| Group                       | Count | Source                                      |
| --------------------------- | ----- | ------------------------------------------- |
| CIMTool core modules        | 2     | Eclipse PDE product export `plugins/` directory |
| Kena/lib vendored JARs      | 9     | `Kena/lib/*.jar`                            |
| CIMUtil/lib vendored JARs   | 12    | `CIMUtil/lib/*.jar`                         |

If a JAR file is not found at its expected path, the script prints `[MISSING]` and
skips it rather than aborting, so you can see all missing files in a single run.

### When to Re-run the Install Script

| Situation | Action Required |
| ------------------------------------------------------------ | ------------------------------------------- |
| First-time project setup | Run install script once after Eclipse PDE export |
| New CIMTool PDE product export — **local dev only** | Re-run install script with the new export root path. For official releases this is handled automatically by `release-cimtool.bat`. |
| Vendored JARs in `Kena/lib/` or `CIMUtil/lib/` updated | Re-run install script and answer `Y` at the vendor JAR prompt |

### Committing lib-repo/ to Source Control

The `lib-repo/` directory **should be committed to source control**. This means any
developer who clones the repository can immediately run `mvn package` without needing
to run the install script themselves, as long as the committed `lib-repo/` contents
are current. The install script only needs to be re-run when the underlying JARs change.

> **Release context:** In the release workflow, `release-cimtool.bat` invokes
> `install-jars.bat` automatically, which may update `lib-repo/` if Kena or CIMUtil
> versions have changed. Any resulting changes to `lib-repo/` must be committed back
> to master via Pull Request **after** the build completes and **before** the release
> tag is created. See the **Phase 4 — Post-Build and Publishing** section in
> `CIMToolProduct/CIMToolProduct-README` for the full sequencing details.



## Building

### Local development build

**Step 1 — Populate `lib-repo/` after each PDE product export:**
```
cd cimtool-cli
install-jars.bat <path-to-PDE-export>
```

**Step 2 — Build the uber JAR:**
```
cd cimtool-cli
mvn clean package
```

The uber JAR will be written to:

```
cimtool-cli/target/cimtool-cli.jar
```

The `dist/` directory is **not** populated by a local build. It is only updated by
`CIMToolProduct/release-cimtool.bat` during the official release process, which
signs the JAR before copying it there. Use `target/cimtool-cli.jar` directly for
local development and testing.


### Official release build

For official releases, the entire pipeline is orchestrated automatically by
`CIMToolProduct/release-cimtool.bat`. That script handles all of the following
in sequence:

1. Signs all plugin JARs in the CIMToolProduct PDE export
2. Signs `CIMTool.exe`
3. Populates `lib-repo/` by invoking `install-jars.bat`
4. Builds the `cimtool-cli` uber JAR via Maven
5. Signs `cimtool-cli.jar`
6. Deploys the signed JAR to `dist/` and generates its SHA-256 checksum
7. Packages the signed CIMToolProduct export into the distribution ZIP
8. Generates the ZIP SHA-256 checksum

Do not run `mvn clean package` directly for official releases. See
`CIMToolProduct/CIMToolProduct-README` for the full release procedure.

### Builder Types

`--list-builders` output includes a type indicator in parentheses for each builder. There are four types:

| Type | Description |
| ---- | ----------- |
| `TEXT` | XSLT-based; produces text output with indentation post-processing applied |
| `XSD` | XSLT-based; produces XSD schemas with XML Schema validation of the output |
| `TRANSFORM` | XSLT-based; produces XML output without additional post-processing |
| `JAVA` | Java-based; uses CIMTool generator classes directly — no XSLT stylesheet involved |

The `JAVA` builders (`xml`, `ttl`, `simple-flat-owl`, `simple-owl`, `legacy-rdfs`, `simple-flat-owl-augmented`, `simple-owl-augmented`, `legacy-rdfs-augmented`) were previously executable only within the Eclipse desktop application. Beginning with the 2.3.0 release they are fully supported by the CLI.

### Common Usage Examples

**Minimum valid invocation — process all profiles using their configured builders:**
```
java -jar cimtool-cli.jar --project-dir .\MyProject
```
```
java -jar cimtool-cli.jar -pd .\MyProject
```

The CLI reads project settings from `.\MyProject`, processes every `.owl` file in `.\MyProject\Profiles`, and for each profile runs whichever builders are configured on it. Generated artifacts are written to `.\MyProject\Profiles` alongside the source `.owl` files — the same default behavior as the desktop application.

**Transform a single profile using a named builder:**
```
java -jar cimtool-cli.jar ^
  --project-dir .\MyProject ^
  --profile .\MyProject\Profiles\MyProfile.owl ^
  --builder xsd ^
  --output .\output
```
```
java -jar cimtool-cli.jar -pd .\MyProject -p .\MyProject\Profiles\MyProfile.owl -b xsd -o .\output
```

**Transform all profiles with an explicit output directory:**
```
java -jar cimtool-cli.jar ^
  --project-dir .\MyProject ^
  --output .\output
```
```
java -jar cimtool-cli.jar -pd .\MyProject -o .\output
```

**Apply a custom XSLT stylesheet to a single profile:**
```
java -jar cimtool-cli.jar ^
  --project-dir .\MyProject ^
  --profile .\MyProject\Profiles\MyProfile.owl ^
  --xslt .\my-custom-transform.xsl ^
  --output-ext json ^
  --output .\output
```
```
java -jar cimtool-cli.jar -pd .\MyProject -p .\MyProject\Profiles\MyProfile.owl -x .\my-custom-transform.xsl -oe json -o .\output
```

**Apply bundled default copyright templates:**
```
java -jar cimtool-cli.jar ^
  --project-dir .\MyProject ^
  --copyright-defaults
```

**Apply custom copyright text from files:**
```
java -jar cimtool-cli.jar ^
  --project-dir .\MyProject ^
  --copyright-multi-line .\copyright-multi.txt ^
  --copyright-single-line .\copyright-single.txt
```

**List all available builders with their type indicators and output extensions:**
```
java -jar cimtool-cli.jar --list-builders
```
```
java -jar cimtool-cli.jar -l
```

**Print Java stack trace on fatal error:**
```
java -Dcimtool.debug=true -jar cimtool-cli.jar --project-dir .\MyProject
```

**Enable verbose diagnostic logging:**
```
java -Dlogback.configurationFile=.\logback-debug.xml -jar cimtool-cli.jar --project-dir .\MyProject
```

> **Note:** When multiple builders are flagged on a profile, each builder is executed independently. A failure in one builder is logged with the builder name and full stack trace, and processing continues with the remaining builders. A summary of failed builders is printed at the end of each profile's processing run.

## Logging


### Production defaults

The CLI uses SLF4J with Logback for all diagnostic output. Under normal operation
the root log level is `WARN`, which means only genuine warnings and errors are
written to the console. All log output is directed to `stderr` so it does not
contaminate structured `stdout` output.

Under normal operation the CLI is intentionally quiet. Only schema integrity
errors (missing range types, invalid cardinality declarations, invalid package
references) and JDBC connection failures will appear.


### Enabling diagnostic logging

A `logback-debug.xml` configuration file is distributed in `dist/` alongside
`cimtool-cli.jar`. Place both files in the same directory and activate the debug
configuration with:

```
java -Dlogback.configurationFile=./logback-debug.xml -jar cimtool-cli.jar [options...]
```

This overrides the production `logback.xml` bundled inside the JAR without
modifying it.

The debug configuration enables the following:

| Logger package | Level | What it covers |
| --- | --- | --- |
| `au.com.langdale.xmi` | `TRACE` | Per-attribute and per-class resolution detail in `parseAttributes()` and `parseClasses()`; CIM model interpretation stages in `CIMInterpreterImpl`; datatype inference in `UMLInterpreter`; JDBC connection errors in `EAPParser` and `QEAParser` |
| `au.com.langdale.profiles.cl` | `DEBUG` | `CLISchemaParser` schema import events (orphaned classes, unused types) suppressed at `WARN` in production |
| `net.ucanaccess`, `org.hsqldb` | `ERROR` | UCanAccess and HSQLDB reserved-word noise remains suppressed |
| root | `DEBUG` | All other CIMTool packages |


### Capturing log output to a file

Log output (warnings, errors, debug messages) is written to `stderr` and progress
messages to `stdout`. The commands below capture both streams together.

**Linux / macOS** — writes to file and shows output on screen simultaneously:
```
java -Dlogback.configurationFile=./logback-debug.xml -jar cimtool-cli.jar [options...] 2>&1 | tee debug.log
```

**Windows Command Prompt** — writes to file only (no simultaneous screen output):
```
java -Dlogback.configurationFile=.\logback-debug.xml -jar cimtool-cli.jar [options...] > debug.log 2>&1
```

**Windows PowerShell** — writes to file and shows output on screen simultaneously:
```
java -Dlogback.configurationFile=.\logback-debug.xml -jar cimtool-cli.jar [options...] 2>&1 | Tee-Object -FilePath debug.log
```


### Overriding the log configuration

Any logback configuration file can be substituted at runtime without rebuilding
the JAR:

```
java -Dlogback.configurationFile=/path/to/your/logback.xml -jar cimtool-cli.jar [options...]
```

The `logback-debug.xml` in `dist/` can be used as a starting point. Adjust logger
names and levels to focus on specific subsystems.



## Technical Notes

### Builder Resource Layout

`BuilderRegistry` loads builder configurations and XSLT stylesheets at runtime via classpath resource lookups. Two configuration files are loaded:

- `builders.json` — XSLT-based builder catalog (`TEXT`, `XSD`, `TRANSFORM` types), loaded via `getResourceAsStream("/builders.json")`
- `java-builders.json` — Java-based builder catalog (`JAVA` type), loaded via `getResourceAsStream("/java-builders.json")`

Both files are loaded into a single unified `TreeMap`, producing one alphabetically sorted registry. XSLT stylesheets are loaded via `getResourceAsStream("/" + xslFileName)`. All of these resources must reside at the **classpath root** of the uber JAR — not under a `builders/` subdirectory.

This is achieved by declaring `CIMUtil/builders/` itself (not its parent) as the
resource root via `build-helper-maven-plugin` in `pom.xml`. This causes Maven to
place `builders.json`, `java-builders.json`, all `*.xsl` files, `includes/`, and `adoc-themes/` directly
at the root of the uber JAR's classpath.

`CIMUtil/import-reports/` is added as a second resource with `targetPath` set to
`import-reports`, preserving the directory structure under that path in the uber JAR.
This matches the `/import-reports/...` resource paths used by the CLI at runtime
(e.g. `/import-reports/puml/sparx-ea-style.puml`, `/import-reports/asciidoc/`).

### Saxon-HE and XSLT Service Registration

Saxon-HE 10.8 registers its `TransformerFactory`, `XPathFactory`, and other XSLT
SPI implementations via `META-INF/services` entries. When multiple JARs contributing
`META-INF/services` files are merged by the Shade plugin, only one file would
normally survive. The `ServicesResourceTransformer` in `pom.xml` correctly merges
all entries so Saxon is discovered at runtime via the standard
`javax.xml.transform` service-loader mechanism. Without this, XSLT processing
would fail at runtime with a `TransformerFactoryConfigurationError`.

### JAR Signature Stripping

Several vendored JARs contain cryptographic signature files (`META-INF/*.SF`,
`META-INF/*.DSA`, `META-INF/*.RSA`). These become invalid once the JAR's contents
are merged into the uber JAR, causing the JVM to throw a `SecurityException` on
startup. The Shade plugin filter in `pom.xml` strips all signature files from every
contributing JAR before assembly.

#### Is This a Security Concern?

No — stripping signatures is not a security concern for this use case, and here
is why.

**What JAR signing actually protects:** JAR signing allows a consumer to verify that
a JAR came from a specific publisher and that its contents have not been tampered
with since signing. The signature is mathematically bound to the exact byte contents
of the original JAR file.

**Why stripping is necessary and unavoidable:** When Maven Shade merges 25 JARs into
one uber JAR, the contents of each contributing JAR are extracted and repacked into
a new ZIP structure. This invalidates every original signature unconditionally —
regardless of whether anything malicious occurred. There is no way to produce a
valid shaded uber JAR that preserves the original signatures. The only two outcomes
are: (1) strip the signatures and the uber JAR works correctly, or (2) leave the
signatures and the JVM throws a `SecurityException` on startup because the signatures
no longer match the repacked contents. Every uber JAR produced by Maven Shade or any
equivalent shading tool strips signatures — this is standard, expected, and
unavoidable.

**What you lose and whether it matters:** Stripping gives up the ability to verify at
runtime that e.g. `saxon-he-10.8.jar` was genuinely published by Saxonica unmodified.
However, that trust decision was already made implicitly when the JARs were added to
the project. Additionally, the JVM security manager — the runtime mechanism that
actually enforced JAR signature verification — was deprecated in Java 17 and removed
in Java 21, meaning runtime signature enforcement no longer exists on Java 20+
regardless. Finally, JAR signing was originally designed to protect sandboxed
environments such as browser applets and Java Web Start applications. Neither of
those threat models applies to a CLI tool run directly from the command line.

**Distribution and signing policy:** `cimtool-cli` is an externally published artifact
intended for public consumption. All officially released builds must be signed using
the UCA Users Group EV certificate on the IdenTrust SafeNet hardware token. Signing
is a release-only operation performed by the CIMTool project lead as part of the
coordinated release process orchestrated by `CIMToolProduct/release-cimtool.bat`.
That script builds the unsigned uber JAR, signs it via PKCS#11, deploys it to
`dist/`, and generates the SHA-256 checksum.

In both cases the assembled uber JAR is signed as a single artifact, replacing the
stripped vendor signatures with a unified signature over the entire JAR.



## Versioning

The uber JAR version tracks the CIMTool bundle version declared in
`CIMUtil/META-INF/MANIFEST.MF`. To update the version, change the `<version>`
element in `pom.xml` to match the new bundle version. The `finalName` in the Shade
plugin configuration picks this up automatically, so the output JAR name will
update accordingly.



## JAR Signing

### Overview

`cimtool-cli` is an externally published, publicly distributed artifact. All
officially released builds must be signed using the UCA Users Group EV certificate
on the IdenTrust SafeNet hardware token. Signing is a release-only operation
performed by the CIMTool project lead — it is not part of the normal developer
build workflow.

For local development and testing, the unsigned uber JAR produced by
`mvn clean package` is fully functional. The JVM does not require a JAR to be
signed in order to execute it via `java -jar`.

### Release Signing

Signing of `cimtool-cli.jar` for official releases is handled automatically by
`CIMToolProduct/release-cimtool.bat`. That script orchestrates the complete
release pipeline for both CIMToolProduct and cimtool-cli — it invokes
`install-jars.bat`, runs `mvn clean package`, signs the resulting uber JAR via
PKCS#11 using the IdenTrust EV hardware token, deploys the signed JAR to `dist/`,
and generates the `dist/cimtool-cli.jar.sha256` checksum file.

See `CIMToolProduct/CIMToolProduct-README` for the full release signing procedure,
prerequisites, and the complete step-by-step walkthrough.

### Verifying the Release Signature

To verify the signed JAR use Eclipse Temurin's `jarsigner` (not Zulu or other
OpenJDK distributions — see the CIMToolProduct-README Prerequisites section for
the reason):

```
"C:\Program Files\Eclipse Adoptium\jdk-20.0.2.9-hotspot\bin\jarsigner.exe" ^
  -verify -verbose -certs dist\cimtool-cli.jar
```

The key result to look for is **"jar verified"** at the top of the output. This
confirms the signature is cryptographically valid and the JAR has not been tampered
with since signing.

### Timestamping

The signing script embeds a trusted RFC 3161 timestamp from IdenTrust's TSA server
(`http://timestamp.identrust.com`). This ensures the signature remains verifiable
indefinitely — even after the EV signing certificate expires. See
`CIMToolProduct/CIMToolProduct-README` for a full explanation of how timestamped
signatures work and why they remain valid after certificate expiry.
