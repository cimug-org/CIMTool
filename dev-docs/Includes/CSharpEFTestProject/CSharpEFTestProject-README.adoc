
== **CSharpEFTestProject**

A self-contained smoke test harness that validates the architecture and design decisions
embedded in C# EF Core entity code generated by CIMTool's `csharp-ef-rdfs.xsl` builder,
using a real relational EF Core provider backed by an in-memory SQLite database.


=== Overview

CSharpEFTestProject exercises the generated `SampleProfile.csharp-ef-rdfs.cs` file against
two overarching goals that guided the design of the `csharp-ef-rdfs.xsl` XSLT builder.
The first goal is that application developers should write as little EF Core and Fluent API
code as possible — the generated `DbContextBase` class provides DbSet properties, complete
Fluent API model configuration, a `DbContextOptions` constructor for dependency injection
integration, and automatic compound orphan cleanup through `SaveChanges` and
`SaveChangesAsync` overrides, leaving the developer with nothing more than a one-class thin
subclass. The second goal is that the companion SQL DDL script produced by the
`sql-rdfs-ansi92.xsl` builder is the authoritative schema, and the generated EF Core code
is a tightly complementary layer above it — for example, the SQL DDL's reverse-reference
`ON DELETE CASCADE` pattern for compound value objects is paired with
`DeleteBehavior.Restrict` in the EF Core model configuration, enforcing the compound
ownership invariant at both layers.

The harness runs as a single-file top-level-statements console application. Each test
section executes against a freshly created in-memory SQLite database scoped to that
section, ensuring complete isolation between sections. The generated C# file is referenced
directly from the CIMTool project output directory rather than being copied, so every
test run reflects the current builder output.


=== Project Structure

[source]
----
CSharpEFTestProject/
├── CSharpEFTestProject/          CIMTool project directory
│   └── Profiles/
│       ├── SampleProfile.csharp-ef-rdfs.cs    Generated EF Core entities (linked by test)
│       ├── SampleProfile.rdfs-ansi92.sql       Generated SQL DDL (read by SQL parity tests)
│       └── SampleProfile.xml                   CIMTool profile source
└── EfCoreSmokeTest/              C# smoke test project
    ├── EfCoreSmokeTest.csproj
    ├── Program.cs                              All test sections and assertion helpers
    ├── SampleProfileDbContext.cs               Thin hand-written subclass of DbContextBase
    └── GeneratedOnlySampleProfileDbContext.cs  Baseline context without SaveChanges cleanup
----

`SampleProfileDbContext` subclasses the generated `SampleProfile.DbContextBase` and
configures SQLite. It is the thin subclass pattern that all application developers are
expected to follow.

`GeneratedOnlySampleProfileDbContext` subclasses `DbContext` directly, wires
`ModelConfiguration` manually, and provides no `SaveChanges` overrides. It exists
specifically to prove the contrast: without the generated `DbContextBase` cleanup logic,
replaced and detached compound graphs are left orphaned in the database. This baseline
context is intentionally kept incomplete.


=== Dependencies on Other Projects

The `EfCoreSmokeTest` project has a `<Compile Include>` item that links
`SampleProfile.csharp-ef-rdfs.cs` directly from the CIMTool project's `Profiles`
directory. No copy of the generated file is maintained inside the test project. The
`SampleProfile.rdfs-ansi92.sql` file is read at runtime by the SQL parity test section
from the same `Profiles` directory.

=== Test Architecture

The harness uses a `RunSection(string name, Action action)` helper that calls each section
function and records its name on success. Any `AssertCondition` failure inside a section
throws `InvalidOperationException`, which terminates the run with a descriptive message
identifying the failed assertion. Sections are not individually isolated from each other
at the process level, but each section that requires a database creates a fresh in-memory
SQLite connection and database via `WithFreshDatabase`, `WithFreshDatabaseAsync`, or
`WithFreshGeneratedOnlyDatabase`, ensuring no state leaks between sections.

Certain behaviors are documented through a `diagnostics` list rather than hard
`AssertCondition` calls. Diagnostics appear in the output under a `Diagnostics:` heading
after the section summary. They are used for intentional architectural divergences between
the SQL DDL and EF Core layers — specifically the `PARITY GAP OBSERVED` message confirming
that the SQL `ON DELETE CASCADE` and EF Core `DeleteBehavior.Restrict` plus `SaveChanges`
cleanup are complementary rather than identical — and for the `BASELINE OBSERVED` message
confirming that orphan accumulation without `DbContextBase` is the expected baseline
behavior of `GeneratedOnlySampleProfileDbContext`.


=== Test Coverage

==== Reflection Contract

Verifies that the generated C# file carries the correct data annotation attributes without
consulting the EF Core model. Checks `[Key]`, `[Column]`, and `[MaxLength]` on surrogate
and natural key properties across compound types, enumerated types, and entity classes.
Verifies that `[Index(IsUnique = true)]` is present on all compound FK properties of
`Organisation`. Confirms that `DbContextBase` exposes a `DbSet<T>` property for every
Tier 3 entity class and that no compound type or enumerated type DbSet is publicly exposed.
Verifies that `DbContextBase` declares a protected constructor accepting `DbContextOptions`,
enabling standard ASP.NET Core dependency injection registration.

==== EF Core Metadata Contract

Consults the EF Core model via `context.Model` to verify table names, primary key
properties, relational column names, delete behaviors, unique index presence, and foreign
key principal targets. Confirms `DeleteBehavior.Restrict` for all compound FK properties
and `DeleteBehavior.ClientNoAction` for all independent `IdentifiedObject` association FK
properties. Verifies that all four enumerated types register `Name` as their primary key
in the EF Core model, confirming alignment with the `PRIMARY KEY` declaration in the
companion SQL DDL. Uses EF Core metadata APIs rather than reflection to verify the model
as registered, not just as annotated.

==== SQL Schema Parity

Reads the generated `SampleProfile.rdfs-ansi92.sql` file and verifies key DDL patterns by
string matching. Confirms that compound FK columns carry `UNIQUE` constraints in the SQL
DDL, that reverse-reference `ON DELETE CASCADE` constraints are present for compound
cleanup, and that the `Name.IdentifiedObject` index is emitted. Adds a `PARITY GAP
OBSERVED` diagnostic to document the intentional difference between the SQL DDL's
cascade-based cleanup and the EF Core layer's `DeleteBehavior.Restrict` plus `SaveChanges`
override approach.

==== Generated C# Text Integrity

Reads the generated `SampleProfile.csharp-ef-rdfs.cs` file as text and verifies that key
structural markers are present: the banner comment, entity class declarations, the
`DbContextBase` class, the `DeleteOrphanedCompounds` method, and both
`DeleteBehavior.ClientNoAction` and `DeleteBehavior.Restrict` references. Confirms that
removed sample types are absent from the generated output.

==== Lookup Equality Semantics

Verifies that generated `Equals` and `GetHashCode` implementations behave correctly for
all three entity categories. Enumerated types compare by natural `Name` key. Compound types
compare by surrogate `Id`. `IdentifiedObject` subclasses compare by `MRId` and reject
cross-type equality even when `MRId` values match. Confirms that compound constructors
assign a non-empty surrogate `Id` on construction without requiring application code to
set it, and that each constructor call produces a distinct `Id` — enforcing the
one-compound-per-slot ownership invariant that the generated cleanup logic depends on.

==== Name Association Behavior

Exercises the full insert, load, and delete lifecycle of the `Name → IdentifiedObject`
association. Verifies that a `Name` row can be inserted with a navigation property
reference, that `IdentifiedObject` loads correctly via `Include`, and that deleting a
`Name` leaves the referenced `Organisation` and its `IdentifiedObject` base row intact.
Verifies that attempting to delete an `Organisation` still referenced by a `Name` row
fails with `DbUpdateException`, confirming that `DeleteBehavior.ClientNoAction` is
correctly enforced by the database FK constraint.

==== Parent Organisation Delete Guard

Verifies that `DeleteBehavior.ClientNoAction` on the `Organisation.ParentOrganisationId`
foreign key correctly blocks deletion of a `ParentOrganization` row that is still
referenced by a child `Organisation`. Confirms that both rows remain in the database after
the failed delete.

==== Generated Compound Replacement Cleanup

Verifies that replacing `Phone1` and `StreetAddress` with new compound instances in a
single `SaveChanges` call causes the generated `DbContextBase` cleanup to delete the
original orphaned compound rows. Also verifies that the Phase 2 nested compound expansion
correctly cleans up `Status`, `StreetDetail`, and `TownDetail` rows orphaned when their
parent `StreetAddress` is replaced.

==== Generated Null Detach Cleanup

Verifies that nulling out `ElectronicAddress`, `Phone1`, and `StreetAddress` FK properties
causes the generated `DbContextBase` cleanup to delete all orphaned compound rows,
including nested `Status`, `StreetDetail`, and `TownDetail` rows whose parent
`StreetAddress` was detached.

==== Generated Compound Async Cleanup

Mirrors the compound replacement test using `await context.SaveChangesAsync()` throughout.
Verifies that the `SaveChangesAsync` overrides in the generated `DbContextBase` correctly
invoke `CollectCompoundOrphans` and `DeleteOrphanedCompounds`, including Phase 2 nested
compound expansion, producing the same cleanup guarantees as the synchronous path.

==== Generated Compoundless Entity Cleanup

Inserts, modifies, and deletes an `Organisation` with no compound properties assigned,
exercising the early-return path in the generated `SaveChanges` overrides. When
`CollectCompoundOrphans` returns an empty list the second `base.SaveChanges()` call is
skipped and the row count from the first save is returned directly. Verifies that this
path completes correctly and that the full TPT deletion chain removes both the
`Organisation` row and its `IdentifiedObject` base row cleanly.

==== Generated Mapping Baseline

Exercises the same compound replacement and null-detach operations using
`GeneratedOnlySampleProfileDbContext`, which has no `SaveChanges` overrides. Confirms that
all replaced and detached compound rows — including nested compounds — remain orphaned in
the database when cleanup is absent, establishing the baseline that motivates the generated
`DbContextBase` implementation.

==== Inheritance Storage

Inserts `Organisation`, `ParentOrganization`, `WireInfo`, and `OverheadWireInfo` entities
and queries the base `IdentifiedObjects` DbSet. Verifies that EF Core's TPT
(Table-Per-Type) implementation correctly materializes all four rows as their concrete
runtime types when queried through the hierarchy root.

==== Inheritance Delete Cleanup

Deletes `ParentOrganization` and `OverheadWireInfo` entities through their typed DbSet
properties and verifies that all TPT rows — including base `IdentifiedObject` rows and
intermediate table rows — are removed across the full inheritance chain.


=== Eclipse IDE Integration

CIMTool developers who work primarily in Eclipse IDE have two options for accessing the
`EfCoreSmokeTest` project from within their existing Eclipse workspace.


==== Option A — General Project Import (view only)

Importing the project as a General Project requires no additional plugins and takes less
than a minute. It gives full access to all project files — including this README, the
`.csproj`, and all `.cs` source files — through Eclipse's file browser and text editor,
with no C#-specific language intelligence.

. In Eclipse, select *File → Import → General → Existing Projects into Workspace*.
. Set the root directory to the `EfCoreSmokeTest` folder.
. Click *Finish*.

The project appears in the Package Explorer. Files open in Eclipse's built-in text editor.
All developmental work — editing `Program.cs`, running `dotnet run`, debugging — is
performed externally in Visual Studio Code or a terminal. Eclipse serves purely as a
project viewer under this option.


==== Option B — Eclipse aCute Plugin (full C# language support)

Eclipse aCute provides a rich C# editor with error reporting, hover, content assist, and
jump to references using OmniSharp, and syntax highlighting using TextMate grammar.
Integration with OmniSharp-Roslyn and the Language Server Protocol is powered by Eclipse
LSP4E. This is the same OmniSharp language server that the C# extension for Visual
Studio Code uses, so the editing experience — completions, diagnostics, hover
documentation, go-to-definition, find references — is equivalent to working in VS Code,
presented through the familiar Eclipse IDE shell.

aCute supports Eclipse versions through 2025-03 (4.35) and later. The .NET 8 SDK
must already be installed and available on `PATH` before configuring aCute.

*Prerequisites*

* Eclipse IDE (2023-06 or later recommended)
* .NET 8 SDK installed and on `PATH` — verify with `dotnet --version`

*Installation steps*

. Open Eclipse and select *Help → Eclipse Marketplace*.
. Search for `aCute`.
. Select *Eclipse aCute: C# development tools* and click *Install*.
. Accept the license agreement and restart Eclipse when prompted.

*Verifying the installation*

Open any `.cs` file from the `EfCoreSmokeTest` project. Eclipse should open the file
using the Generic Editor with C# language features active. If OmniSharp does not start
automatically, check *Window → Preferences → aCute* to confirm the OmniSharp server
path is detected. OmniSharp requires the .NET SDK on `PATH` to locate and load the
project's dependencies; if it fails to start, verify `dotnet --version` is resolvable
from a terminal launched from Eclipse's working directory.

*Note for CIMTool developers*

aCute is an optional convenience for developers who prefer to remain in Eclipse while
reviewing or making minor edits to the C# smoke test. Running the test (`dotnet run`)
and managing NuGet packages still requires a terminal or Visual Studio Code. For
significant changes to `Program.cs` or the project structure, Visual Studio Code remains
the recommended environment.


=== Running the Test

The .NET 8 SDK must be installed and `dotnet` available on `PATH`.

[source,powershell]
----
cd EfCoreSmokeTest
dotnet restore
dotnet run
----

A successful run prints a summary of completed sections followed by any informational
diagnostics. Any assertion failure terminates the run immediately with a message
identifying the failed condition. No test runner or xUnit dependency is required — the
harness is a self-contained console application.


=== Relationship to Other Projects

`CSharpEFTestProject` depends directly on the output of the CIMTool
`csharp-ef-rdfs.xsl` builder. The generated `SampleProfile.csharp-ef-rdfs.cs` file is
compiled into the test project via a linked `<Compile Include>` item; every builder change
is reflected immediately in the next `dotnet run` without any copy step. The SQL parity
section reads `SampleProfile.rdfs-ansi92.sql` generated by the `sql-rdfs-ansi92.xsl`
builder to verify that the EF Core model and the authoritative SQL DDL remain aligned.
The project has no dependency on the JPA builder or any other CIMTool builder output.
