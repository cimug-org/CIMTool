= cimtool-cli

A standalone Maven project that assembles the CIMTool command-line interface into
a single self-contained executable uber JAR (`cimtool-cli.jar`) using the
Maven Shade plugin.

No Eclipse, OSGi runtime, or workspace is required to run the resulting JAR.



== Overview

This project is a *packaging-only* Maven module. It contains no Java source of
its own. Its sole purpose is to:

. Collect `kena.jar` and `cimutil.jar` from a CIMTool Eclipse PDE product export
. Collect all vendored third-party JARs from `Kena/lib/` and `CIMUtil/lib/`
. Embed the XSLT builder stylesheets from `CIMUtil/builders/` at the classpath root
. Merge everything into a single executable uber JAR via Maven Shade

The CLI entry point is `au.com.langdale.profiles.cl.CIMToolCLI`, which lives in
`CIMUtil/src/`.

=== Eclipse PDE Product Export Must Run First

This project sources `kena.jar` and `cimutil.jar` from a CIMTool Eclipse PDE product
export. These JARs are *not* generated by Maven and are *not* taken from the Eclipse
workspace `bin/` directories — they must come from a full PDE product export.

To perform the export in Eclipse:

. From the Eclipse menu select *File > Export*
. In the Export dialog select *Plug-in Development > Eclipse product* and click *Next*
. Set the fields as shown in the screenshot below:
** *Configuration:* `/CIMToolProduct/CIMTool.product`
** *Root directory:* `CIMTool-<version>` (e.g. `CIMTool-2.3.0`) —
   replace `2.3.0` with the actual official release version being built
** *Destination:* Select *Directory* and set the export path
   (e.g. `D:\CIMTool-Releases`)
** Check *Synchronize before exporting*
** Check *Allow for binary cycles in target platform*
. Click *Finish*

image::{docdir}/readme-images/Eclipse_Product_Export.png[Eclipse Product Export dialog,700]

After the export completes, the `plugins\` directory under the export root will
contain versioned plugin folders including:

----
<export-root>\plugins\au.com.langdale.kena_<version>\kena.jar
<export-root>\plugins\au.com.langdale.cimutil_<version>\cimutil.jar
----

The install script locates these JARs automatically by searching for plugin folders
matching `au.com.langdale.kena_*` and `au.com.langdale.cimutil_*` and extracts the
version number from the plugin folder name (e.g. `au.com.langdale.kena_3.3.4` → `3.3.4`).
This means the script works for any release version without modification.

NOTE: The version extracted from the plugin folder name must match the `<version>`
declared for `kena` and `cimutil` in `pom.xml`. Update `pom.xml` when the
version changes.

== Dependency Management — Local File Repository

=== Why a Local Maven File Repository?

All of this project's dependencies are vendored JARs that live under `Kena/lib/`
and `CIMUtil/lib/` in the Eclipse PDE project tree, plus `kena.jar` and
`cimutil.jar` sourced from the CIMTool Eclipse PDE product export. None of these are published to Maven Central.

Maven's `system` scope (the most obvious way to reference local JARs) is formally
deprecated as of Maven 3.8 and causes a false-positive validation error in Eclipse's
embedded Maven (m2e):

----
'dependencies.dependency.systemPath' for com.hp.hpl.jena:jena:jar must specify
an absolute path but is ${project.basedir}/../Kena/lib/jena-2.6.3.jar
----

To avoid this, the project uses a *local file repository* instead. A `lib-repo/`
directory inside `cimtool-cli/` is declared as a standard Maven repository in
`pom.xml`:

[source,xml]
----
<repositories>
    <repository>
        <id>local-lib-repo</id>
        <url>file://${basedir}/lib-repo</url>
    </repository>
</repositories>
----

All dependencies are then declared as normal Maven dependencies with no `system`
scope or `systemPath` — m2e resolves them cleanly against `lib-repo/` with no
errors or warnings.

=== Populating lib-repo/

The `lib-repo/` directory is populated by running the provided install script, which
calls `mvn install:install-file` for each JAR and places it into `lib-repo/` using
the standard Maven repository layout (`groupId/artifactId/version/artifact-version.jar`).

*Passing the export root as an argument:*
----
cd cimtool-cli
install-jars.bat D:\CIMTool-Releases\CIMTool-2.3.0
----

*Path contains spaces — wrap in double quotes:*
----
cd cimtool-cli
install-jars.bat "D:\My Releases\CIMTool-2.3.0"
----

*Omit the argument to be prompted interactively:*
----
cd cimtool-cli
install-jars.bat
----

When run without an argument, the script will prompt:
----
  No export root specified.
  Enter the path to the CIMTool Eclipse PDE product export root.
  Example: D:\CIMTool-Releases\CIMTool-2.3.0
  Tip: If the path contains spaces, wrap it in double quotes.

  Export root: _
----

After installing the two core module JARs, the script prompts whether to
also install the vendor JARs:

----
  The vendor JARs (Kena\lib and CIMUtil\lib) rarely change between
  releases. Re-installing them is only needed if new or updated JARs
  have been added to those directories since the last run.
  If unsure, answer Y to ensure lib-repo is fully up to date.

  Install vendor JARs? (Y/N): _
----

=== Committing lib-repo/ to Source Control

The `lib-repo/` directory *should be committed to source control*. This means any
developer who clones the repository can immediately run `mvn package` without needing
to run the install script themselves, as long as the committed `lib-repo/` contents
are current. The install script only needs to be re-run when the underlying JARs change.



== Building

Once a CIMTool PDE product export has been performed and the install script has
been executed at least once, building the uber JAR is a single command:

----
cd cimtool-cli
mvn clean package
----

The uber JAR will be written to:

----
cimtool-cli/target/cimtool-cli.jar
----

To build, sign, verify, and deploy to the `dist/` directory in a single command:

----
cd cimtool-cli
mvn clean install -Psign
----

The signed JAR will be written to both `target/` and `dist/`:

----
cimtool-cli/target/cimtool-cli.jar
cimtool-cli/dist/cimtool-cli.jar
----

=== Common Usage Examples

*Transform a single profile using a named builder:*
----
java -jar cimtool-cli.jar ^
  --project-dir .\MyProject ^
  --profile .\MyProject\profiles\MyProfile.owl ^
  --builder xsd ^
  --output .\output

java -jar cimtool-cli.jar ^

  --profiles-dir .\MyProject\profiles ^

----

*Use a custom XSLT stylesheet:*
----
java -jar cimtool-cli.jar ^
  --project-dir .\MyProject ^
  --profile .\MyProject\profiles\MyProfile.owl ^

----
java -jar cimtool-cli.jar --list-builders
----

*Show version:*
----
java -jar cimtool-cli.jar --version
----

The version is read at runtime from the JAR manifest and reflects the Maven
project version embedded at build time (e.g. `CIMTool CLI version 2.3.0`).

*Enable debug output on error:*
----
java -Dcimtool.debug=true -jar cimtool-cli.jar ...

== Technical Notes

=== XSLT Builders Resource Layout

`BuilderRegistry` loads `builders.json` and all XSLT stylesheets via
`getResourceAsStream("/builders.json")` and `getResourceAsStream("/" + xslFileName)`,
meaning these resources must reside at the *classpath root* of the uber JAR — not
under a `builders/` subdirectory.

This is achieved by declaring `CIMUtil/builders/` itself (not its parent) as the
resource root via `build-helper-maven-plugin` in `pom.xml`. This causes Maven to
place `builders.json`, all `*.xsl` files, `includes/`, and `adoc-themes/` directly
at the root of the uber JAR's classpath.

`CIMUtil/import-reports/` is added as a second resource with `targetPath` set to
`import-reports`, preserving the directory structure under that path in the uber JAR.
This matches the `/import-reports/...` resource paths used by the CLI at runtime
(e.g. `/import-reports/puml/sparx-ea-style.puml`, `/import-reports/asciidoc/`).

=== Saxon-HE and XSLT Service Registration

Saxon-HE 10.8 registers its `TransformerFactory`, `XPathFactory`, and other XSLT
SPI implementations via `META-INF/services` entries. When multiple JARs contributing
`META-INF/services` files are merged by the Shade plugin, only one file would
normally survive. The `ServicesResourceTransformer` in `pom.xml` correctly merges
all entries so Saxon is discovered at runtime via the standard
`javax.xml.transform` service-loader mechanism. Without this, XSLT processing
would fail at runtime with a `TransformerFactoryConfigurationError`.

=== JAR Signature Stripping

Several vendored JARs contain cryptographic signature files (`META-INF/*.SF`,
`META-INF/*.DSA`, `META-INF/*.RSA`). These become invalid once the JAR's contents
are merged into the uber JAR, causing the JVM to throw a `SecurityException` on
startup. The Shade plugin filter in `pom.xml` strips all signature files from every
contributing JAR before assembly.

==== Is This a Security Concern?

No — stripping signatures is not a security concern for this use case, and here
is why.

*What JAR signing actually protects:* JAR signing allows a consumer to verify that
a JAR came from a specific publisher and that its contents have not been tampered
with since signing. The signature is mathematically bound to the exact byte contents
of the original JAR file.

*Why stripping is necessary and unavoidable:* When Maven Shade merges 25 JARs into
one uber JAR, the contents of each contributing JAR are extracted and repacked into
a new ZIP structure. This invalidates every original signature unconditionally —
regardless of whether anything malicious occurred. There is no way to produce a
valid shaded uber JAR that preserves the original signatures. The only two outcomes
are: (1) strip the signatures and the uber JAR works correctly, or (2) leave the
signatures and the JVM throws a `SecurityException` on startup because the signatures
no longer match the repacked contents. Every uber JAR produced by Maven Shade or any
equivalent shading tool strips signatures — this is standard, expected, and
unavoidable.

*What you lose and whether it matters:* Stripping gives up the ability to verify at
runtime that e.g. `saxon-he-10.8.jar` was genuinely published by Saxonica unmodified.
However, that trust decision was already made implicitly when the JARs were added to
the project. Additionally, the JVM security manager — the runtime mechanism that
actually enforced JAR signature verification — was deprecated in Java 17 and removed
in Java 21, meaning runtime signature enforcement no longer exists on Java 20+
regardless. Finally, JAR signing was originally designed to protect sandboxed
environments such as browser applets and Java Web Start applications. Neither of
those threat models applies to a CLI tool run directly from the command line.

*Distribution and signing policy:* `cimtool-cli` is an externally published artifact
intended for public consumption. All officially released builds *must* be signed.
The signing model follows a two-tier approach:

* *Development and internal use:* Every member of the CIMTool development team signs
  their local builds using a personal self-signed certificate generated via
  `create-keystore.bat`. Self-signed certificates are sufficient for development,
  testing, and internal validation purposes.

* *Official releases:* When a formal release of CIMTool is packaged for distribution,
  `cimtool-cli.jar` must be signed using the UCAIug organisation's official
  certificate and keystore. Access to this keystore is restricted to the CIMTool
  project lead, who is solely responsible for final release-level packaging and
  signing. See the <<official-release-signing,Official Release Signing>> section
  for the release signing procedure.

In both cases the assembled uber JAR is signed as a single artifact, replacing the
stripped vendor signatures with a unified signature over the entire JAR.



== Versioning

The uber JAR version tracks the CIMTool bundle version declared in
`CIMUtil/META-INF/MANIFEST.MF`. To update the version, change the `<version>`
element in `pom.xml` to match the new bundle version. The `finalName` in the Shade
plugin configuration picks this up automatically, so the output JAR name will
update accordingly.



== JAR Signing

=== Overview

`cimtool-cli` is an externally published, publicly distributed artifact. All builds
intended for distribution *must* be signed. The `sign` Maven profile handles signing
via the `maven-jarsigner-plugin`. The build is designed so that swapping from a
self-signed certificate to the UCAIug official certificate requires no changes to
`pom.xml` — only the keystore file and properties file need to change.

Signing follows a two-tier model:

[cols="1,3", options="header"]
|===
| Context
| Certificate to Use

| Local development and internal testing
| Personal self-signed certificate (generated via `create-keystore.bat`)

| Official release packaging
| UCAIug organisation certificate — project lead only
|===

=== Development Signing — Self-Signed Certificate

All members of the CIMTool development team should sign their local builds using a
personal self-signed certificate. This is sufficient for development, testing, and
internal validation.

==== Step 1 — Generate a Self-Signed Keystore

Run the provided script to create a keystore at `%USERPROFILE%\.cimtool\cimtool-keystore.jks`:

----
create-keystore.bat
----

The script prompts for a keystore password, key password, and Distinguished Name
details (organisation, city, country, etc.). The certificate is generated with a
10-year validity period.

[cols="1,1", options="header"]
|===
| Keystore Question
| Recommended Response

| Keystore password:
| <password>

| What is your first and last name?^1^
| www.ucaiug.io

| What is the name of your organizational unit?
| Software Development

| What is the name of your organization?
| UCA International Users Group (UCAIug)

| What is the name of your City or Locality?
| Shell Knob

| What is the name of your State or Province?
| Missouri

| What is the two-letter country code?
| US

| Password for <cimtool-cli>:^2^
| <password>
|===

^1^ Although `keytool` labels this field "first and last name", it actually maps
to the *Common Name (CN)* field in the certificate's Distinguished Name. For a
code-signing certificate associated with an organisation rather than an individual,
it is conventional to use a domain name or application identifier here instead of
a personal name. `www.ucaiug.io` is recommended for all UCAIug-associated builds.
Developers generating a personal self-signed certificate for local development may
use their own name or any meaningful identifier in this field.

^2^ This is the password for the specific key entry identified by the alias
`cimtool-cli`. For a self-signed development keystore it is safe and conventional
to reuse the keystore password here. The two passwords only need to differ in
higher-security environments where multiple keys share the same keystore. This is
the case for the official UCAIug keystore but not for a personal self-signed keystore.

WARNING: Avoid the following special characters in keystore passwords: `&`, `$`, `%`,
`^`, `<`, `>`. These are interpreted by the Windows command shell (`cmd.exe`) as
control characters and will cause the `mvn install -Psign` command to fail with a
cryptic error. Letters, digits, and characters such as `#`, `@`, `!`, `-`, `_` are
all safe to use.

==== Step 2 — Create the Signing Properties File

Copy the template to your `.cimtool` folder and fill in the values:

----
copy cimtool-signing.properties.template %USERPROFILE%\.cimtool\cimtool-signing.properties
----

Edit `%USERPROFILE%\.cimtool\cimtool-signing.properties` and set:

----
signing.keystore=C:/Users/YourName/.cimtool/cimtool-keystore.jks
signing.alias=cimtool-cli
signing.storepass=<keystore password chosen during keystore creation>
signing.keypass=<key password chosen during keystore creation>
----

NOTE: Use forward slashes (`/`) in the `signing.keystore` path, not backslashes.
Java `.properties` files treat backslash (`\`) as an escape character, so a path
like `C:\Users\...` will be misread. Forward slashes work correctly on Windows.

IMPORTANT: `cimtool-signing.properties` contains passwords and must *never* be
committed to source control. It lives outside the repository by design.

==== Step 3 — Build and Sign

----
mvn clean package -Psign
----

Maven will assemble the uber JAR, sign it, verify the signature, then copy the
signed JAR to `dist/`. The artifact is written to both locations:

----
cimtool-cli/target/cimtool-cli.jar
cimtool-cli/dist/cimtool-cli.jar
----

==== Verifying the Signature Manually

----
jarsigner -verify -verbose target/cimtool-cli.jar
----

[[official-release-signing]]
=== Official Release Signing — UCAIug Certificate

When packaging a formal CIMTool release for public distribution, `cimtool-cli` must
be signed using the UCAIug organisation's official certificate and keystore. Access
to this keystore is restricted to the CIMTool project lead, who is solely responsible
for final release-level packaging and signing.

NOTE: The steps below are for the project lead only. Development team members use
their personal self-signed certificate as described above.

==== Step 1 — Obtain the UCAIug Keystore

Obtain the official UCAIug keystore file and its credentials through the appropriate
internal channel. Store the keystore file securely — it must never be committed to
source control or shared outside the release process.

==== Step 2 — Configure the Signing Properties File

Create or update `%USERPROFILE%\.cimtool\cimtool-signing.properties` with the
UCAIug keystore details:

----
signing.keystore=<absolute path to UCAIug keystore file>
signing.alias=<UCAIug certificate alias>
signing.storepass=<UCAIug keystore password>
signing.keypass=<UCAIug key password>
----

==== Step 3 — Build and Sign the Release JAR

----
mvn clean package -Psign
----

Maven will assemble the uber JAR, sign it with the UCAIug certificate, verify the
signature, then copy the signed JAR to `dist/`. The artifact is written to both
locations:

----
cimtool-cli/target/cimtool-cli.jar
cimtool-cli/dist/cimtool-cli.jar
----

The JAR in `dist/` is the release-ready artifact for inclusion in the official
CIMTool release distribution.

==== Step 4 — Verify the Release Signature

----
jarsigner -verify -verbose -certs target/cimtool-cli.jar
----

The `-certs` flag prints the certificate chain, confirming the JAR is signed by
the UCAIug organisation certificate and not a self-signed development certificate.

The key result to look for is *"jar verified"* at the top of the output. This
confirms the signature is cryptographically valid and the JAR has not been tampered
with.

NOTE: When verifying a JAR signed with the self-signed development certificate,
jarsigner will report two warnings that are expected and harmless:

----
Warning:
This jar contains entries whose certificate chain is invalid. Reason: PKIX path
building failed: unable to find valid certification path to requested target
This jar contains entries whose signer certificate is self-signed.
----

These warnings are inherent to self-signed certificates. The first indicates there
is no chain back to a trusted CA — because the certificate is its own root. The
second simply restates that fact. Neither indicates a problem with the JAR or the
signing process. Both warnings will disappear once the JAR is signed with the
official UCAIug CA-issued certificate, which chains back to a trusted root that
the JVM already knows about.

=== Timestamping

The signing configuration embeds a trusted timestamp from DigiCert's free public TSA
server (`http://timestamp.digicert.com`). This ensures the signature remains
verifiable even after the signing certificate expires — important for both self-signed
development certificates and the UCAIug release certificate. If building in an
offline environment, remove the `<tsa>` element from the `maven-jarsigner-plugin`
configuration in `pom.xml`.
